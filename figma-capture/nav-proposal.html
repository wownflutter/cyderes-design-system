Cyderes Platform Navigation Proposal

Internal UX Architecture — Unifying MDR, Exposure Management, and Asset Management

March 2026

This document presents a unified navigation architecture for the Cyderes platform. The goal is to collapse three historically siloed products — MDR (Detect & Respond), Exposure Management, and Asset/Identity Inventory — into a single coherent experience. The Issue Map graph visualization serves as the cross-domain connective tissue, enabling analysts to traverse domain boundaries through direct object interaction rather than menu switching. Below you will find the proposed object model, navigation structure, page flows, graph placement strategy, and consistency standards.

⚙ Settings
Three domains as peers, not tabs
Detect & Respond, Exposure, and Inventory appear as equal top-level sections in the sidebar. No domain is "primary" — they are parallel entry points into the same underlying data model. This avoids the current pattern where Exposure feels bolted onto an MDR tool.
Issue Map is top-level — the cross-domain connective tissue
The Issue Map sits outside any domain section, separated by dividers. This signals that it is not owned by MDR or Exposure — it belongs to the platform. It is the primary mechanism for traversing domain boundaries through direct graph interaction.
Object detail pages are NOT nav items
Asset Detail, Identity Detail, Incident Detail, and Finding Detail are reached through drill-down (clicking a table row or list item) or through graph node interaction (clicking a node in the Issue Map). They do not appear in the sidebar because they are contextual destinations, not persistent views.
Inventory unifies Assets + Identities as platform objects
Rather than scattering asset lists across MDR and Exposure, the Inventory section owns all entity-management views. Assets, Identities, and Environments are platform-level objects that every domain references. This prevents duplicate asset lists and ensures a single source of truth.

4 Graph Placement Rules
Summary
When: Dashboard pages (Command Center, Exposure Dashboard)
Where: Hero zone below KPI strip, full width
Variant: Environment
"Orient the analyst — where are the problems?"
GRAPH — Summary
Investigation
When: Entity detail pages (Asset, Identity, Incident)
Where: Primary content zone, entity-centered
Variant: Asset-Centered or Identity-Centered
"Support the investigation — what's connected, what's at risk?"
GRAPH — Investigation
Exploration
When: Issue Map full page (top-level nav)
Where: Full viewport canvas
Variant: Unscoped or focused on a specific entity
"Free-form investigation — trace paths across all domains"
GRAPH — Exploration (full canvas)

The graph is the primary mechanism for cross-domain movement. Clicking nodes traverses domain boundaries invisibly — the analyst sees connected objects regardless of which product "owns" them. Breadcrumbs record the journey, enabling the analyst to retrace steps across MDR, Exposure, and Inventory without losing context.
6 Consistency Rules
Rule Standard
Page anatomy Breadcrumb → Header → Filters → KPI Strip → Content → Sidebar
Entity header Avatar + Name + Type badge + Metadata row + Risk score — consistent across all detail pages
Graph visualization Issue Map component only — delete all other graph variants. One component, three modes (Summary, Investigation, Exploration).
Data tables Shared component: severity pills, sortable columns, filter chips, pagination. No per-domain table variants.
Entity cards Standardized layout: type icon + entity name + risk badge + permissions indicator. Same card on every list page.
Filter bar Severity pills + domain filter toggles + search input + "Add Filter" button. Identical behavior everywhere.
Action bar Primary (emerald) + Secondary (ghost/outline) + Danger (red). Button hierarchy consistent across all pages.
